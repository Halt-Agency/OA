Intended Audience
This document is intended for technically trained developers. Please ensure you share this with developers who intend to implement the API. Support will not be provided for non technically trained users.

Introduction
The Tracker API is built on HTTP and is RESTful. It has predictable resource URLs and returns HTTP response codes to indicate errors.

It returns JSON in the HTTP body. Use the HTTP/REST library in your favorite programming language when using this API, or generate a client SDK using your preferred development IDE.

Many Tracker actions are supported by APIs which are documented and available for use, so that you may automate any necessary workflows or processes. This API reference documentation contains the most commonly integrated resources.

Authentication
All calls to this API require an authorised JWT token, which is obtained by exchanging a valid OAuth2 token for a JWT token using the /api/auth/exchangetoken endpoint.

The token is acquired with the OAuth2 authentication flow using our OAuth2 API.

Authentication steps

Acquire a token using the OAuth2 flow with the /oauth2/authorize and /oauth2/token endpoints
See the OAuth2 API documentation for details on these endpoints and note in particular that they are published on a separate domain to this API
Exchange your token for a JWT token using the /api/auth/exchangetoken endpoint
Send your JWT token in the Authorization header with all subsequent calls to the API
Important: JWT tokens expire, so you should always check for an authentication failure (401) response and obtain a new JWT token by calling the /api/auth/exchangetoken endpoint, if necessary.

Errors
Tracker follows the error response format proposed in RFC 9457, which is also known as Problem Details for HTTP APIs.

As with any API responses, your client must be prepared to gracefully handle additional members of the response.

Rate Limits
Tracker implements rate limiting on the API. Presently around 100 requests per minute.

If the rate limit is exceeded, the API returns a '429 Too Many Requests' response.

Response Codes
This API follows REST specification. For convenience below is a table of returned response codes:

Code	Meaning
200 OK	The request was successful.
201 Created	A new resource has been successfully created.
202 Accepted	The request has been accepted for processing, but the processing is not complete.
204 No Content	The request was successful, but there is no content to return.
400 Bad Request	The request was malformed or invalid. Check the request parameters or body.
401 Unauthorized	Authentication is required or has failed. Verify your JWT token.
403 Forbidden	The client does not have permission to access the requested resource.
404 Not Found	The requested resource could not be found.
405 Method Not Allowed	The HTTP method is not supported for the requested resource.
409 Conflict	The request could not be completed due to a conflict with the current state of the resource.
415 Unsupported Media Type	The request entity has a media type that the server does not support.
422 Unprocessable Entity	The server understands the content type and syntax, but the content is semantically invalid.
429 Too Many Requests	The client has sent too many requests in a given amount of time (rate limiting).
500 Internal Server Error	A generic error occurred on the server.
502 Bad Gateway	The server received an invalid response from an upstream server.
503 Service Unavailable	The server is currently unavailable, usually due to maintenance or overload.
These status codes are returned in the HTTP response status line, and error responses may include a body formatted according to RFC 9457 (Problem Details) for additional context.
